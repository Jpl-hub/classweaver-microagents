# ClassWeaver 2026 升级改造路线图

## 1. 目标重定义

当前仓库具备可运行的产品骨架，但整体仍偏向课程设计级 Demo。2026 版升级的目标不是“把功能堆更多”，而是把项目升级为一个可在简历、面试、作品集里完整讲清楚的 AI 教辅系统：

- 有稳定的工程交付能力
- 有可解释、可评估的 RAG 与 agent 工作流
- 有能支撑“学习效果提升”叙事的数据闭环
- 有清晰的版本控制、提交记录和演进路径

## 2. 现状判断（以代码为准）

### 已有基础

- Django + DRF 后端 API 已成型
- Vue 3 + Vite 前端可作为交互壳继续演进
- 已有预习任务、知识库、测验、推荐、时间线等核心实体
- 已有 `planner -> rewriter -> tutor` 基础流水线
- 已有基础测试目录，可扩展为完整回归体系

### 当前主要短板

- 后台任务使用 Python 线程，缺少可靠队列、重试和观测
- 检索只有本地 FAISS dense 路线，未形成 hybrid retrieval / rerank / citation 闭环
- agent 输出仍以弱结构 JSON 为主，缺少严格 schema 校验和回归评测
- 推荐和 tutor 逻辑偏模板化，学习状态建模不足
- 前端交互和信息架构仍偏 demo 化，缺少专业产品感
- 自动化测试、CI、部署、日志追踪尚不足以支撑“工程项目”叙事

## 3. 升级原则

- 不重写，不轻易推翻已有骨架
- 优先做能显著提升项目可信度和简历价值的改造
- 先建立评测与观测，再做 prompt、模型和工作流优化
- 代码默认追求直接、可验证、易回归，不为了“稳”而堆大量兜底分支
- 每个阶段完成后都保留可演示版本，并提交到 GitHub

## 4. 分阶段改造计划

## Phase 0: 基线审计与开发闭环

### 目标

让项目重新进入“随时可以启动、测试、提交”的状态。

### 交付物

- 依赖、运行方式、环境变量、数据库、前端构建的基线审计
- 开发环境说明统一
- 本地一键启动方案
- 测试基线跑通
- 第一版升级任务板

### 任务

- 核对后端、前端依赖是否需要升级到当前稳定线
- 补齐 Python、Node、数据库、向量存储的本地运行说明
- 清理 README 与实际代码不一致部分
- 让 `pytest`、前端构建、最基本接口检查可执行
- 建立分支、提交、推送规范

### 完成标志

- 新环境按文档可启动
- 基础测试可执行
- GitHub 上有独立升级分支

## Phase 1: 工程化重构

### 目标

把系统从“单机 demo 流程”升级为“具备可靠执行能力的后端服务”。

### 交付物

- 后台任务队列替换线程方案
- 配置分层与环境隔离
- 更稳定的数据持久化和索引管理
- 更清晰的模块边界

### 任务

- 用任务队列替代 `threading.Thread`
- 为任务状态、失败原因、重试次数、耗时增加可观测字段
- 统一配置读取和默认值策略
- 对 pipeline 输入输出建立显式 schema
- 收敛 API 层、service 层、agent 层职责

### 技术建议

- 优先考虑 `PostgreSQL + pgvector`
- 队列优先考虑轻量、可维护方案；如需引入 Redis，再一起纳入 compose
- 保留 FAISS 作为本地快速模式，不再作为唯一生产型方案

## Phase 2: RAG 升级

### 目标

把当前“能检索”升级为“检索质量可解释、可量化、可优化”的 RAG 系统。

### 交付物

- hybrid retrieval
- reranking
- 引用回填与证据展示
- 检索评测集与离线指标

### 任务

- 升级文档切分、元数据设计、索引写入逻辑
- 引入 query rewrite 和失败回退策略
- 增加 dense + sparse 或 dense + keyword 的 hybrid 检索
- 增加 reranker
- 回答结果绑定 chunk 级 citation
- 构建小型离线评测集，覆盖命中率、证据引用率、拒答质量

### 简历亮点方向

- “实现面向教育内容的可解释 RAG”
- “构建检索与生成双阶段评测体系”

## Phase 3: Agent Workflow 升级

### 目标

把当前串行流水线改造成可解释、可观测、可演进的 workflow。

### 交付物

- 结构化 agent 节点定义
- 可回放 trace
- 显式路由和失败策略
- prompt 版本管理

### 任务

- 重构 planner / rewriter / tutor 的职责边界
- 把“是否检索、是否重写、是否生成测验、是否给 hint”做成显式决策
- 记录每一步模型、输入摘要、输出摘要、耗时、错误信息
- 建立 prompt regression 测试

### 简历亮点方向

- “设计多阶段 agent workflow，并支持链路追踪与回归评估”

## Phase 4: 学习闭环与推荐升级

### 目标

让系统从“给一组题和建议”升级为“能基于行为数据给出下一步学习动作”的产品。

### 交付物

- 错题诊断卡
- 知识点掌握画像
- tutor hint 机制
- recommendation 与 timeline 联动

### 任务

- 按知识点维度累计掌握度
- 将 quiz 结果直接反馈给 tutor
- 让 recommendation 不再只是静态模板，而是基于诊断输出
- 将 timeline 事件与学习动作连接起来

### 简历亮点方向

- “构建基于知识点掌握度的个性化学习推荐”

## Phase 5: 前端产品化升级

### 目标

让项目达到可投递、可演示、可录视频的视觉和交互质量。

### 交付物

- 重构首页、结果页、知识库页、课堂教练页
- 更清晰的信息架构
- 可视化引用、错题、推荐、时间线
- 更规范的状态管理和请求层

### 任务

- 梳理页面主路径和任务工作台
- 将结果页做成“知识点 / 证据 / 测验 / 建议”一体化界面
- 优化 loading、轮询或流式反馈
- 统一样式变量和组件层级

## Phase 6: 观测、评估、部署

### 目标

形成一套能支撑面试深聊的质量保障体系。

### 交付物

- 请求链路追踪
- 模型调用统计
- 离线评测脚本
- 部署文档与演示环境

### 任务

- 接入 tracing / prompt 日志
- 建立模型、prompt、检索策略对比实验
- 形成 benchmark 结果表
- 准备线上演示环境或标准化本地演示方案

## 5. Git 与提交策略

- 主分支保持可用，不直接在 `main` 上做高风险开发
- 升级阶段统一在 `codex/*` 或后续专题分支推进
- 每完成一个可验证子目标就提交一次
- 优先提交“能跑通、能测试、能回归”的改动，不堆超大杂糅提交
- 当本机 GitHub 凭据可用时，及时推送远端，保留阶段演进记录

## 6. 近期执行顺序

### Sprint 1

- 跑通本地环境
- 补齐测试依赖
- 校正文档与实际代码偏差
- 建立升级分支并推送
- 输出详细任务拆分

### Sprint 2

- 后台任务线程替换
- 配置和 schema 重构
- 第一版日志与 trace

### Sprint 3

- RAG 检索升级
- 引用与评测闭环

## 7. 需要用户侧提供的资源

以下资源并非现在全部阻塞，但越早准备越好：

- 可用的模型 API Key
- 一个确定的数据库目标方案
- 如果走队列方案，是否接受引入 Redis
- 如果要做线上演示，目标部署平台
- 如果要做教育效果评测，后续需要少量样例资料与问答集

## 8. 当前默认策略

在用户没有额外限制前，默认采用以下方向推进：

- 保留 Django + DRF，不做后端框架重写
- 保留 Vue，但升级前端工程化和界面质量
- 先做工程化和 RAG 质量，再做更激进的 agent 改造
- 所有阶段都以“简历价值 + 面试可讲清楚”为评价标准
