# 2026 评测与反思设计

## 目标

ClassWeaver 在 2026 的核心差异不应该是“也能把资料塞进大模型”，而应该是：

- 能解释内容为什么可信
- 能量化学习输出哪里薄弱
- 能反思下一轮该补检索、补题目、还是补多模态证据

当前实现已经把这三层接入主链路：

- `RetrievalDiagnostics`
  记录 `pgvector / hybrid / rerank / final_hits / source_counts`
- `evaluation.rule_metrics`
  用硬规则计算引用覆盖、题目完整度、练习支持度
- `evaluation + reflection`
  用 LLM judge 输出 verdict、风险、缺失证据、学习体验判断和系统改进建议
- `review_summary`
  当 reflection 触发补检索/重生成时，显式记录 review cycle 的前后分数和检索扩展

## 现有评测结构

### 1. 过程评测

不是只看最终回答，而是拆分成：

- 检索是否足够
- 知识点是否有 refs
- 题目是否有 refs 和 variants
- tutor 练习是否有 citations
- 学习承接是否足够

### 2. 结果评测

当前输出包含：

- `evaluation.scores`
  `groundedness / citation_coverage / quiz_quality / tutoring_value / learner_fit / overall`
- `evaluation.rule_metrics`
  硬规则计数、比例、检索闸门
- `reflection`
  `diagnosis / next_actions / should_regenerate / should_expand_retrieval / should_add_multimodal_review`

### 3. 页面可视化

首页和历史页已经能展示：

- RAG 诊断卡
- 评测与反思卡
- Review cycle 前后分数变化
- 模型追踪卡

## 与论文的对应关系

### Self-RAG

关键词：self-reflection、critique、按需检索。

对应设计：

- 不把检索当作必然正确输入
- 在 `reflection.should_expand_retrieval` 中显式输出是否该补检索
- 后续可把 `reflection.should_regenerate` 作为二次生成触发条件

### CRAG

关键词：先评估检索质量，再决定 corrective action。

对应设计：

- `RetrievalDiagnostics` 先看召回和融合情况
- `rule_metrics.gates.needs_more_retrieval` 作为纠错闸门
- 后续可继续把 web search / query rewrite / doc filtering 接进 corrective path

### RAGChecker

关键词：RAG 评测应拆 retrieval 和 generation，不只看最终答案。

对应设计：

- 过程指标独立暴露
- benchmark 支持 retrieval 报告、citation 报告和两份报告 diff
- evaluator 继续把 generation 风险结构化输出

### MathTutorBench / TutorBench

关键词：会做题不等于会教；hint、feedback、active learning 要单独评。

对应设计：

- 单独评 `tutoring_value / learner_fit`
- learner experience 不只看“对不对”，还看 `smoothness / cognitive_load / personalization`
- 后续 recommendation 应与错因、followup 和 hint generation 联动

### MMTutorBench

关键词：OCR 和多模态链路会显著影响 tutoring quality。

对应设计：

- 当前先保留 `should_add_multimodal_review`
- 后续如果引入图像/PDF 图示理解，可把 slide figure、板书截图、题目配图作为二次复核证据

### PreFlect

关键词：reflection 不只事后复盘，还要事前预判。

对应设计：

- 当前 evaluator 主要是 post-hoc audit
- 下一阶段应把 reflection 前移到 planner 前或 planner 后，做 pre-execution critique

## 下一阶段建议

### A. 让反思真正驱动重试

把下面三类动作做成正式策略：

- `should_expand_retrieval=true`
  触发 query rewrite / top_k 扩展 / corrective retrieval
- `should_regenerate=true`
  触发 planner 或 tutor 定向重生成
- `should_add_multimodal_review=true`
  触发图示/OCR/图片证据复核

当前版本已经完成第一步：

- 当 reflection 要求补检索或重生成时，会执行一轮显式 review cycle
- review 不是静默兜底，而是把触发原因、扩展后的 `top_k`、分数变化、最终 verdict 全部保留
- 当需要补检索时，会先执行一次 query rewrite，再重新拉取证据
- 当问题主要在练习层时，会优先走 `tutor_only`，而不是无脑整条链路重跑

### B. 做学习体验级 benchmark

新增三组数据集：

- 错因诊断集
- hint 质量集
- recommendation 命中集

### C. 加用户状态闭环

把以下信号接进 recommendation：

- quiz 错因
- tutor followup 完成情况
- timeline 执行记录
- 历次 evaluation / reflection 结果

## 需要服务器资源时再做的升级

这些方向很值钱，但会明显增加部署和推理成本：

- 多模态 slide / figure / handwritten note 理解
- cross-encoder 或 MLLM reranker
- 基于学生历史轨迹的个性化 reward model
- 专门的 tutor judge / reward model 微调

等进入阿里云演示部署阶段时，再统一做资源设计。
